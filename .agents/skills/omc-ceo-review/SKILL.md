# OMC 기능 범위 재검토 (CEO 관점)

> **이 스킬을 쓰면 안 되는 상황**:
> - 이미 구현 완료된 기능 사후 검토 → `$omc-retro` 사용
> - 버그 수정처럼 범위가 명확한 작업 → 바로 `$omc-task`
> - 요구사항이 아직 불명확 → `$omc-brainstorm` 먼저

---

## Step 0: 컨텍스트 수집

> **AI는 아래 커맨드를 직접 실행하고 결과를 확인한 후 모드를 선택한다. 건너뛰지 않는다.**

```bash
python3 scripts/omc.py state status --target . 2>/dev/null
git log --oneline -5 2>/dev/null
```

수집 결과 연결:
- `omc.py state status` → 현재 세션/작업 컨텍스트 파악

---

## 모드 선택

> **모드가 명시되지 않으면 AI가 아래 4가지를 보여주고 선택을 요청한다. 자동으로 HOLD를 적용하지 않는다.**

| 모드 | 언제 사용 |
|---|---|
| `EXPAND` | 현재 범위가 너무 좁아서 임팩트가 작을 때 |
| `SELECTIVE` | 일부 기능만 선택적으로 구현할 때 |
| `HOLD` | 지금 만들어야 하는지 확신이 없을 때 |
| `REDUCE` | 범위를 줄여서 빠르게 검증하고 싶을 때 |

호출 예: `$omc-ceo-review REDUCE 로그인 기능`

## 공통 검토 질문 5개 (모든 모드 필수)

> **각 질문은 "예/아니오" 한 줄 답변으로 끝내지 않는다. 구체적 수치 또는 근거를 반드시 포함한다.**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CEO REVIEW
대상: [기능 이름]   모드: [EXPAND/SELECTIVE/HOLD/REDUCE]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[공통 검토]
1. 이 기능이 없으면 사용자가 이탈하는가?
   (근거: 유사 사례, 사용자 피드백, 데이터 등)
   → _______________

2. 핵심 가치 제안을 강화하는가?
   강화하지 않는다면 왜 지금 만드는가? (대안 검토 여부 포함)
   → _______________

3. 단순한 버전으로도 80%의 가치를 낼 수 있는가?
   단순 버전이란: _______________
   → _______________

4. 성공 지표 1개 (측정 가능한 수치):
   → _______________

5. 배포 시한 안에 완료 가능한가?
   불가능하다면 범위를 어떻게 줄일 수 있는가?
   (※ 보안·인프라·기술 부채 등 필수 작업은 시간 기준 대신 위험 평가로 대체 가능)
   → _______________
```

## 모드별 추가 질문 (해당 모드 선택 시 필수)

### EXPAND 모드 추가 질문
```
E1. 현재 범위 대비 어떤 기능이 추가되면 임팩트가 2배 이상 커지는가?
    (구체적으로 어떤 기능인지, 왜 지금 포함해야 하는지)
    → _______________

E2. 확장 시 개발 비용은 얼마나 증가하는가?
    임팩트 대비 비용이 정당한가?
    → _______________
```

### SELECTIVE 모드 추가 질문
```
S1. 선택적으로 구현했을 때, 제외된 기능 없이도 완결성 있는 사용자 경험이 가능한가?
    → _______________

S2. 포함/제외 기준은 무엇인가? 그 기준이 이번 한 번에만 적용되는 임시 기준인가?
    → _______________
```

### HOLD 모드 추가 질문
```
H1. 지금 하지 않으면 6개월 후에도 여전히 필요한가?
    그때 하는 것과 지금 하는 것의 차이는?
    → _______________

H2. 대기 비용(기회비용)은 무엇인가? 지금 안 하면 놓치는 것이 있는가?
    → _______________
```

### REDUCE 모드 추가 질문
```
R1. 줄인 버전에서 무엇을 포기하는가? 그 포기가 허용 가능한가?
    → _______________

R2. 줄인 후에도 Q4의 성공 지표를 측정할 수 있는가?
    → _______________
```

## 이후 액션

| 결론 | 다음 단계 |
|---|---|
| APPROVED | → `$omc-plan` 으로 설계 진입 |
> ⛔ **AI는 위 결과를 출력한 뒤 반드시 멈춘다. 사용자가 다음 스킬을 명시적으로 호출하기 전까지 자동으로 진입하지 않는다.**

| REDUCE | 범위 축소 후 이 스킬 재실행 → APPROVED 확인 후 `$omc-plan` |
| REJECT | 보류 → `.omc/notepad.md` 에 사유 기록 |
