# OMC Critique — 냉정한 비판 리뷰

> **이 스킬의 역할**: 칭찬하지 않는다.
> 계획이나 코드가 "왜 틀렸는지", "무엇을 놓쳤는지", "어떤 가정이 틀릴 수 있는지"만 찾는다.
> `omc-review`(품질 체크)나 `omc-office-hours`(제품 사고)와 다르다 — 이건 **Pre-mortem**이다.

---

## 실행 순서

### Step 0: 컨텍스트 자동 감지

> **AI는 아래 커맨드를 직접 실행하고 결과를 확인한 후 모드를 결정한다. 건너뛰지 않는다.**

> **untracked 파일 주의**: `git diff`는 staged/tracked 변경만 잡는다.
> 새로 만든 파일이 `.gitignore`에 포함되거나 아직 add 안 됐으면 diff에 안 나타난다.
> 이 경우 PLAN 모드로 빠지지만 실제로는 CODE 모드가 더 정확하다.
> → 아래 추가 확인 후 모드를 결정한다.

```bash
# 코드 변경 여부 확인 (tracked 변경)
git diff --stat HEAD 2>/dev/null | head -20

# untracked 신규 파일 확인 — .gitignore 제외 파일도 포함
git ls-files --others --exclude-standard 2>/dev/null | head -20

# .gitignore 대상이라도 최근 수정된 파일 확인
find . -newer .git/index -not -path './.git/*' \( -name '*.md' -o -name '*.py' -o -name '*.ts' \) 2>/dev/null | head -10

# 플랜 파일 여부 확인
ls .omc/tasks/ 2>/dev/null
cat .omc/notepad.md 2>/dev/null

# 현재 세션 확인
python3 scripts/omc.py state status --target .
```

수집 결과 연결:
- `omc.py state status` → 현재 세션 상태 파악 → PLAN/CODE 모드 판단
- `git diff --stat HEAD` → 변경 범위 파악 → CODE 모드 여부 판단

감지 결과에 따라 모드를 자동 선택한다:
- `git diff`에 내용 있음 → **CODE 모드**
- untracked 신규 파일이 있고 방금 만든 것이면 → **CODE 모드** (git diff에 안 잡혀도)
- `.omc/tasks/` 또는 `.omc/notepad.md`에 계획 있음 → **PLAN 모드**
- 사용자가 텍스트로 계획/전략을 직접 붙여넣음 → **PLAN 모드**
- 대상이 `.md`, 스킬 파일, 설정 파일 등 코드가 아닌 문서 → **PLAN 모드**
- 둘 다 있음 → 사용자에게 선택 요청
- **git diff 없고 untracked도 없는데 "방금 만든 것들"이라고 하면** → `.gitignore` 제외 파일 가능성 → `find` 결과로 CODE 모드 판단

## 이 스킬을 쓰면 안 되는 상황

- 방금 critique를 완료하고 즉시 재실행 → 중복 critique, 시간 낭비
- 범위가 이미 확정되어 있고 품질만 체크하면 충분한 경우 → `$omc-review` 사용
- 단순 버그 수정 (원인이 명확한 경우) → `$omc-investigate` 사용

---

## 행동 원칙 (Critique 스킬 고유 규칙)

1. **칭찬 문장 금지** — "잘 설계됐습니다", "좋은 접근입니다" 같은 문장은 출력하지 않는다.
2. **가정 명시** — 내가 내린 판단의 근거를 반드시 명시한다. 근거 없는 비판 금지.
3. **대안 제시 의무** — 문제를 지적하면 반드시 개선 방향 1개 이상을 함께 제시한다.
4. **범위 준수** — 요청한 계획/코드 범위 밖을 건드리지 않는다.
5. **HOLD 판정 우선** — 확신이 없으면 PROCEED가 아닌 HOLD로 판정한다.
