# /brainstorm — 요구사항 소크라테스식 정제

구현 전에 **"무엇을 만들어야 하는가"** 를 충분히 탐색합니다.
Superpowers의 brainstorming 방식을 OMC에 맞게 적용했습니다.

---

## 언제 쓰나

- 요구사항이 아직 모호하거나 애매할 때
- "이거 만들어줘" 수준의 요청에서 시작할 때
- `/office-hours` 전에 아이디어 자체를 먼저 탐색하고 싶을 때
- 기술 접근법이 여러 개여서 결정이 필요할 때

---

## 브레인스토밍 진행 방식

AI는 **소크라테스 방식**으로 진행합니다 — 답을 먼저 주지 않습니다.
한 번에 질문 1~2개씩, 답변을 듣고 다음 질문을 합니다.

### Phase 1 — 현상 파악 (What)

```
- 지금 어떤 문제가 있나요?
- 이 문제가 없으면 어떤 일이 발생하나요?
- 얼마나 자주, 누가 겪나요?
```

### Phase 2 — 원인 탐색 (Why)

```
- 왜 지금 이 문제가 해결되지 않고 있나요?
- 현재 어떻게 우회하고 있나요?
- 그 우회 방법의 문제점은 무엇인가요?
```

### Phase 3 — 해결 방향 탐색 (How)

```
- 이상적인 해결책을 상상한다면?
- 가장 빠르게 검증할 수 있는 가장 작은 버전은?
- 기술적 제약이 없다면 어떻게 만들겠나요?
```

### Phase 4 — 수렴 (Decide)

충분한 탐색 후 AI가 아래를 정리합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BRAINSTORM 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
핵심 문제: _______________________________________________
근본 원인: _______________________________________________

구현 옵션:
  A) [빠른] _____________________ (예상 시간: ___)
  B) [균형] _____________________ (예상 시간: ___)
  C) [이상적] ___________________ (예상 시간: ___)

권장 옵션: A/B/C — 이유: _______________________________

다음 액션: /office-hours 또는 /plan
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

## 규칙

- AI는 Phase 4 전에 구현 코드를 작성하지 않습니다.
- 사용자가 "그냥 만들어줘"라고 해도 Phase 1~3을 최소 3번은 진행합니다.
- 탐색 결과는 `.omc/notepad.md`에 추가합니다.
