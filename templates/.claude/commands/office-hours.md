# /office-hours — 제품 사고 먼저 (코딩 전)

기능 구현 요청이 들어왔을 때 **코드를 먼저 짜지 않습니다.**

## 질문 답변 후 — AI가 할 일

1. **프레이밍 검토**: 사용자가 요청한 것과 실제 필요한 것이 다를 경우 지적합니다.
   - 예: "로그인 기능이 필요하다고 했지만, 실제 고통은 '팀원 간 작업 상태 공유가 안 됨'입니다. OAuth보다 공유 URL이 더 빠른 해결일 수 있습니다."

2. **범위 도전**: Q4(MVP)가 Q1~Q3(고통/성공기준)을 실제로 해결하는지 검토합니다.

3. **대안 3가지 제시**: 구현 접근법을 최소/중간/이상 3가지로 제안합니다.

4. **설계 문서 초안 작성**: 아래 형식으로 `.omc/notepad.md`에 추가합니다.

```
## [기능명] — Office Hours 결과
날짜: YYYY-MM-DD
대상 사용자: ...
핵심 고통: ...
성공 기준: ...
MVP 범위: ...
비범위: ...
10점 비전: ...
채택한 접근법: ...
```
