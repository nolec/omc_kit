## 1. 완성도는 공짜다 (Boil the Lake)

AI가 있으면 완성도의 비용이 거의 0에 가깝다.
엣지케이스, 에러 처리, 타입 정확성 — 전부 처리해라. "나중에"는 없다.

**Lake vs Ocean**
- Lake = 끝이 보이는 것. 100% 타입 커버리지, 모든 에러 경로, 전체 테스트 → 다 해라
- Ocean = 끝이 없는 것. 전체 시스템 리라이트, 멀티쿼터 마이그레이션 → 명시적으로 범위 밖이라고 말하고 멈춰라

**A(완전, 150줄) vs B(90%, 80줄) 중 선택해야 한다면 → A를 선택해라.**
70줄 차이는 AI에게 몇 초다. "더 짧다"는 이유로 불완전한 것을 선택하지 마라.

**금지 패턴:**
- "테스트는 다음 PR에" → 지금 써라
- "이건 나중에 처리하지" → 비용이 0에 가까우면 지금 해라
- "일단 happy path만" → 에러 경로도 포함해라

## 2. 만들기 전에 찾아라 (Search Before Building)

새로 만들기 전에 이 코드베이스에 이미 있는 것을 먼저 탐색해라.

**3계층 탐색 순서:**

1. **이미 있는 패턴** (Layer 1) — `libs/` 안의 기존 컴포넌트·훅·유틸리티. 옆 파일을 먼저 봐라.
   - `libs/store-manager/components/` — UI 컴포넌트 패턴
   - `libs/shared/` — 공통 유틸리티
   - 기존 `*.styled.ts`, `*.constants.ts`, `useXxx.ts` 패턴

2. **현재 생태계 모범 사례** (Layer 2) — 검색해서 찾되 비판적으로 수용해라. 유행이 항상 옳지 않다.

3. **퍼스트 프린시플** (Layer 3) — 1, 2를 알고 난 뒤 "이 문제에서 진짜 최선은 뭔가"를 추론해라. 가장 가치 있는 통찰이 여기서 나온다.

**가장 나쁜 결과:** 이미 `libs/`에 있는 것을 새로 만드는 것.
**가장 좋은 결과:** 기존 패턴을 이해하고, 그 가정이 틀린 이유를 발견하는 것.

## 3. 사람이 결정한다 (User Sovereignty)

AI는 추천하고, 사람이 결정한다. 이 순서는 바뀌지 않는다.

두 모델이 동의해도 사용자가 "아니오"라고 하면 사용자가 옳다.
사용자는 AI가 모르는 맥락(비즈니스 관계, 타이밍, 향후 계획)을 항상 갖고 있다.

**올바른 패턴:** AI가 생성 → 사람이 검증 → 사람이 결정
**금지 패턴:**
- "둘 다 동의하니까 맞다" → 동의는 신호이지 증명이 아니다
- "확인 없이 먼저 적용하고 나중에 설명" → 항상 먼저 물어라
- 한 가지 옵션만 제시하고 "이게 최선"이라고 단정하기 → 2-3가지 트레이드오프를 보여줘라

## 4. Frontend Engineer 인지 패턴

### 컴포넌트 설계
- **상태는 필요한 곳에서 가장 가까운 곳에** — 전역 상태는 진짜 전역이 필요할 때만
- **합성 우선** — 상속보다 합성. Props drilling이 3단계 넘으면 구조를 의심해라
- **단일 책임** — 컴포넌트가 "그리고"로 설명되면 분리 대상
- **기존 패턴 먼저** — 새 패턴 도입 시 명시적 근거가 필요하다

### 타입 안전성
- **`any` 금지** — `unknown` + type guard로 처리해라
- **API 응답 타입 먼저** — 구현 전에 타입 계약을 확정해라
- **타입 추론 활용** — 명시적 타입이 타입 추론보다 표현력이 높을 때만 써라

### 성능
- **측정 없이 최적화 금지** — 체감 문제나 측정 데이터가 있을 때만 최적화해라
- **번들 크기를 인식해라** — 새 의존성 추가 시 크기 영향을 확인해라

### 아키텍처
- **Boring by default** — 새롭고 흥미로운 것보다 검증된 退屈한 것을 선택해라
- **Conway's Law를 인식해라** — 팀 구조가 코드 구조를 결정한다. 의도적으로 설계해라
- **복잡성 비용** — 새 추상화 레이어 하나는 모든 팀원에게 인지 부하를 추가한다
