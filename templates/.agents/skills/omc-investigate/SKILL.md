# OMC 근본 원인 추적 (4단계 방법론)

**철칙: 수정 전에 반드시 근본 원인을 확인합니다. 증상 패치 금지.**

## 4단계 파이프라인

```
PHASE 1 ▸ ROOT CAUSE INVESTIGATION
현상 (로그/에러/재현 조건): _______________
기대 동작: _______________
☐ 현상 정량화 완료 (추측 없음)

PHASE 2 ▸ PATTERN ANALYSIS
원인 가설 (우선순위 순):
  1. [높음] _______________
  2. [중간] _______________
  3. [낮음] _______________
☐ 가설 생성 완료

PHASE 3 ▸ HYPOTHESIS TESTING
커맨드: _______________
결과: [ ] 확인됨  [ ] 기각됨
☐ 근본 원인 확정됨: _______________

PHASE 4 ▸ IMPLEMENTATION
수정 파일: _______________
☐ 수정 후 검증 PASS
☐ 교훈 기록 완료
```

## 원칙
- 추측 금지 — 데이터/로그/코드 근거 없이 단정하지 않는다
- 증상 패치 금지 — 근본 원인 확인 전 수정 금지
- 3번 연속 실패 → 아키텍처 수준 문제 의심, 범위 재설정
