# OMC 근본 원인 추적 (4단계 방법론)

**철칙: 수정 전에 반드시 근본 원인을 확인합니다. 증상 패치 금지.**

> **이 스킬을 쓰면 안 되는 상황**:
> - 원인이 이미 명확한 버그 수정 → 바로 `$omc-task`
> - 기능 방향 검토 → `$omc-office-hours`

---

## Step 0: 컨텍스트 수집

> **AI는 아래 커맨드를 직접 실행하고 결과를 확인한 후 PHASE 1로 진입한다. 건너뛰지 않는다.**
> 사용자가 에러 로그나 스택 트레이스를 붙여넣었다면 해당 내용도 PHASE 1에 반영한다.

```bash
# 최근 변경 파일 확인
git diff --stat HEAD 2>/dev/null | head -20
git log --oneline -5 2>/dev/null

# 에러 관련 파일 탐색 (에러 키워드를 알고 있을 때)
# rg "[에러 키워드]" --type ts --type tsx -l 2>/dev/null | head -10

# untracked 신규 파일 확인
git ls-files --others --exclude-standard 2>/dev/null | head -10

# 현재 세션/브랜치 확인
python3 scripts/omc.py state status --target . 2>/dev/null
```

수집 결과 연결:
- `omc.py state status` → 현재 작업 컨텍스트 파악
- `git log --oneline` → 최근 변경 맥락 파악

## 4단계 파이프라인

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 1 ▸ ROOT CAUSE INVESTIGATION
현상 (로그/에러/재현 조건): _______________
기대 동작: _______________
재현 불가 시: 로그 추가 후 재현 대기, 또는 간헐적 발생 패턴 기록
```

☐ 현상 정량화 완료 (추측 없음)

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 2 ▸ PATTERN ANALYSIS
원인 가설 (우선순위 순):
  1. [높음] _______________
  2. [중간] _______________
  3. [낮음] _______________

※ 가설이 1개뿐이면 [높음]만 작성해도 된다.
  가설이 4개 이상이면 슬롯을 추가한다.
```

☐ 가설 생성 완료

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 3 ▸ HYPOTHESIS TESTING
검증 커맨드: _______________
결과: [ ] 확인됨  [ ] 기각됨

※ 가설이 모두 기각됐으면:
   → 새 가설을 추가해 PHASE 2로 돌아간다
   → 3회 연속 기각 시 아키텍처 수준 문제로 가정, 범위 재설정

근본 원인 확정: _______________
```

☐ 근본 원인 확정됨

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 4 ▸ IMPLEMENTATION
수정 파일: _______________
핵심 변경: _______________
검증 커맨드: _______________
  (예: npx jest [관련파일], npx tsc --noEmit, 직접 실행 확인)
검증 결과: _______________
```

☐ 수정 후 검증 PASS
☐ 교훈 기록 완료

## 이후 액션

| 상황 | 다음 단계 |
|---|---|
| PHASE 4 완료 | → `$omc-review` 또는 `/ship` |

| 근본 원인이 아키텍처 문제 | → `$omc-ceo-review` 로 범위 재검토 |
