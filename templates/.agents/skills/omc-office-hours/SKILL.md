# OMC 제품 사고 먼저 (코딩 전)

> **brainstorm과의 차이**:
> - `$omc-brainstorm` → "뭘 만들지 모르겠을 때" (방향 탐색)
> - `$omc-office-hours` → "방향은 있지만 제품 관점 검증이 필요할 때" (검증)

> **이 스킬을 쓰면 안 되는 상황**:
> - 요구사항이 아직 모호 → `$omc-brainstorm` 먼저
> - 이미 office-hours 완료 후 바로 재실행 → 불필요

## Step 0: 컨텍스트 수집

> **AI는 아래 커맨드를 직접 실행하고 결과를 확인한 후 질문을 시작한다. 건너뛰지 않는다.**

```bash
python3 scripts/omc.py state status --target . 2>/dev/null
git branch --show-current 2>/dev/null
```

수집 결과 연결:
- `omc.py state status` → 현재 작업 컨텍스트 파악

## 판정 기준

| 판정 | 조건 |
|---|---|
| PROCEED | Q3 성공 기준이 측정 가능 + Q4 MVP 완료 시한을 **사용자가** 답변에 포함 |
| RETHINK | Q1~Q3 중 하나가 불명확 → 다시 답변 |
| HOLD | Q3 측정 불가 또는 Q5 범위가 너무 넓음 → `$omc-ceo-review` |

> ※ 보안·인프라·기술 부채 등 필수 작업은 완료 시한 기준 대신 위험 평가로 대체 가능.
> "2주 내 가능한가?"는 사용자가 판단하며, AI가 임의로 PROCEED를 결정하지 않는다.
