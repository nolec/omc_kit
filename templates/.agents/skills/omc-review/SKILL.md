# OMC 코드 리뷰

이슈를 **치명 / 중대 / 경미 / 제안** 4단계로 분류하고 **파일:라인** 근거를 반드시 첨부합니다.

---

## Step 0: 컨텍스트 자동 수집

> **untracked 파일 주의**: `git diff`는 tracked 변경만 잡는다.
> `.gitignore` 대상이거나 `git add` 전인 신규 파일은 diff에 안 나타난다.
> 아래 명령을 모두 실행한 후 리뷰 범위를 확정한다.

```bash
# 1. tracked 변경 확인
git status -sb
git diff HEAD

# 2. untracked 신규 파일 확인
git ls-files --others --exclude-standard 2>/dev/null | head -20

# 3. .gitignore 제외 파일이라도 최근 수정된 것 확인
find . -newer .git/index -not -path './.git/*' \( -name '*.ts' -o -name '*.tsx' -o -name '*.py' -o -name '*.md' \) 2>/dev/null | head -15

# 4. 리뷰 대상 파일 직접 읽기 (diff로 부족할 때)
# cat <파일경로>

# 5. 세션 상태 확인
python3 scripts/omc.py state status --target . 2>/dev/null
```

**리뷰 범위 확정 규칙**:
- `git diff HEAD` 있음 → staged/unstaged 변경 전체
- untracked/신규 파일 있음 → 해당 파일도 리뷰 범위에 포함
- PR 단위 → `git diff <base>..<head>` 또는 파일 직접 지정
- diff 200줄 이상 → 파일 단위로 나눠서 리뷰

## 이후 액션

| 결과 | 다음 단계 |
|---|---|
| [치명] 이슈 있음 | 수정 후 `$omc-review` 재실행 |
| [중대] 이슈 있음 | 수정 후 검증 커맨드 실행 → 재리뷰 |
| [경미]/[제안]만 있음 | 수정 여부 결정 후 → `/ship` 진행 가능 |
| 이슈 없음 | → `/ship` 진행 |

---

## 이 스킬을 쓰면 안 되는 상황

- 오탈자·주석·공백만 변경된 경우 → 리뷰 불필요, 바로 `/ship`
- 단순 설정값 변경 (상수, env 값) → 리뷰 불필요
- 리뷰 완료 직후 동일 대상 재실행 → 중복, `$omc-critique`로 대체

---

## 행동 원칙

1. **파일:라인 없는 이슈 기술 금지** — "이 부분이 문제입니다" 같은 모호한 지적은 작성하지 않는다.
2. **범위 준수** — 요청한 파일/커밋/PR 범위 밖을 리뷰하지 않는다.
3. **수정 제안 의무** — 이슈를 발견하면 반드시 수정 방향 1개 이상 제시한다.
4. **대용량 diff 분할** — 200줄 이상이면 파일 단위로 나눠서 순서대로 리뷰한다.
5. **칭찬 문장은 최소화** — "잘 됐습니다" 같은 문장을 반복하지 않는다.
