# LLM 운영 규격 (공통 템플릿)

이 문서는 특정 도메인에 종속되지 않는 **공통 LLM 작업 규격**입니다.  
프로젝트별 문서는 이 템플릿을 그대로 복사한 뒤, “프로젝트 계약(정의)”만 채워 사용합니다.

---

## 0) 작업 요청 템플릿 (사용자가 채워서 전달)

아래 **[필수] 4개**가 없으면 LLM은 실행 전에 질문합니다(추측 금지).

- 목표: 무엇을, 왜, 어떤 형태로 만들/고칠지
- 범위: 포함/제외(비범위) 명시 **[필수]**
- 완료 조건(DoD): “무엇이 되면 끝인가”를 테스트/검증 기준으로 **[필수]**
- 제약: 시간/품질/호환성/의존성/권한/네트워크 등 **[필수]**

권장 추가 정보

- 입력 자료: 스샷/로그/샘플 데이터/URL/레퍼런스
- 영향 범위: 사용자/시스템/데이터/비용 영향
- 우선순위: 정확도 vs 속도 vs 비용
- 승인/배포: PR 필요 여부, 릴리즈/배포 방식

---

## 1) 프로젝트 계약(정의) — 불일치의 원천을 제거

동일한 단어가 서로 다른 의미로 쓰이면 재작업이 반복됩니다.  
따라서 아래를 “프로젝트 계약”으로 먼저 확정합니다.

### 1.1 용어/지표/스펙 계약

- 용어 A = (정의)
- 용어 B = (정의)
- 입력 형식 = (예: CSV/JSON, timezone, 단위)
- 출력 형식 = (예: 표/CSV/시트 업데이트/코드 패치)

### 1.2 소스-오브-트루스(우선순위)

예시) `DB > API > 캐시 > UI 표시값` 처럼 충돌 시 어느 쪽을 신뢰하는지.

### 1.3 안전/보안 계약

- 비밀값(토큰/키/쿠키/개인정보)은 **절대 커밋/하드코딩 금지**
- 로그/스크린샷 공유 시 마스킹 규칙(예: 계정ID 뒤 4자리만)

---

## 2) 질문 트리거(멈춰야 할 때)

다음 중 하나라도 해당되면 LLM은 작업을 멈추고 질문합니다.

- 범위/완료조건/제약이 누락되어 “맞게 했는지” 판정이 불가
- 데이터/지표 정의가 여러 해석 가능(예: “유입”, “전환”, “성공”)
- 파괴적 변경(삭제/리셋/마이그레이션/비용 발생)이 필요한데 승인 없음
- 접근 권한/네트워크/의존성 설치가 필요한데 허용 여부 불명

---

## 3) 실행 원칙(효율 + 재현성)

- 최소 변경: 요구된 목적에 필요한 만큼만 변경(불필요한 리팩토링 금지)
- 원인 해결: 증상 패치보다 근본 원인 제거 우선
- 명시적 실패: 필수값 누락/불가능한 상태는 조용히 넘어가지 말고 에러로 드러냄
- 기록 가능성: 결과를 재현할 수 있게 커맨드/환경/입력/출력을 요약

---

## 4) 검증 런북(프로젝트별로 채우기)

아래는 “프로젝트 런북” 섹션으로, 저장소마다 다릅니다.

```bash
# 예시(프로젝트에 맞게 교체)
# setup:
# test:
# lint:
# run:
```

---

## 5) 결과 보고 포맷(LLM 출력 규격)

LLM은 결과를 아래 순서로 보고합니다.

1) 무엇을 했는지(핵심 변경점 3줄 이내)  
2) 어떻게 검증했는지(실행 커맨드/테스트)  
3) 남은 리스크/가정(불확실성)  
4) 다음 액션 1개(가장 임팩트 큰 것)
