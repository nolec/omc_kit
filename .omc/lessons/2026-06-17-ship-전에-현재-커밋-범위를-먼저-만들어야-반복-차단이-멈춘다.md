# ship 전에 현재 커밋 범위를 먼저 만들어야 반복 차단이 멈춘다
날짜: 2026-06-17
태그: ship,scope,state,workflow

## 증상
omc-ship을 반복 실행해도 현재 커밋 범위가 없음 상태가 유지되어 같은 BLOCKED 판정이 계속 반복됐다.

## 원인
staged 된 ship 대상 없이 unstaged dirty 변경만 많은 상태에서 ship 게이트를 먼저 돌려 배포 준비 판단과 범위 정리가 뒤섞였다.

## 적용된 규칙
ship 차단 사유가 현재 커밋 범위 없음이면 ship을 반복하지 말고, 먼저 이번 변경을 최소 커밋 범위로 분리하고 state status로 범위가 생겼는지 확인한 뒤 ship을 다시 실행한다.

## 검증 커맨드
python3 scripts/omc.py state status --target . 에서 현재 커밋 범위가 표시되고 범위 밖 dirty 변경과 분리되는지 확인한 뒤 python3 scripts/omc_tdd_check.py --run-tests 와 omc-ship 게이트를 다시 확인
